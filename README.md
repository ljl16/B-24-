# B站视频搜索实时抓取引擎

一个专门盯住 **B站搜索结果** 的高频抓取脚本。

它不是那种“搜一次就结束”的一次性脚本，而是把 **关键词监控、按天切片、双排序抓取、自动去重、每小时轮询、JSON/XLSX 双落地** 一口气打通，直接把视频搜索监测这件事做成持续运行的采集引擎。

> 如果你要的不是“随便抓一点数据”，而是想把某个关键词在一段时间里的 B站内容变化持续盯住、持续沉淀、持续输出，这个项目就是干这个的。

---

## 这项目到底强在哪

### 1. 每小时自动跑一轮，盯住内容变化

- 启动后先立即执行一轮
- 之后按固定间隔自动继续跑
- 默认每 `1 小时` 执行一轮
- 适合持续监控某个关键词下的新视频变化

### 2. 按天切片抓取，不漏时间区间

- 不是粗暴抓整个时间段
- 而是把起止日期拆成逐天区间
- 每一天都独立请求
- 对时间跨度较长的搜索任务更稳、更清晰、更容易排查

### 3. 双排序同时抓，覆盖“最新”和“最热”

脚本每轮默认同时跑两种排序：

- `pubdate`：最新发布
- `click`：最多播放

这意味着它不是只看“刚发出来的内容”，也不是只看“已经爆了的内容”，而是两个维度一起抓，信息密度直接拉满。

### 4. 自动去重，数据越跑越稳

- 以 `bvid / url / aid` 为核心键去重
- 已经抓过的内容不会反复堆积
- JSON 持续追加但不重复污染
- 导出 XLSX 时再次按 `keyword + bvid` 去重

### 5. 直接产出可用结果，不需要二次整理

输出结果直接落地成：

- `bilibili_search_result.json`
- `bilibili_search_result.xlsx`

抓完就能看，抓完就能筛，抓完就能交付。

## 项目文件说明

### `合并.py`

主程序，负责：

- 配置关键词
- 配置起止日期
- 配置轮询间隔
- 请求 B站搜索接口
- 生成 WBI 签名

用于辅助验证 `w_rid / wts` 签名逻辑的脚本，方便在 Node 环境下调试签名是否一致。

实时累计抓取结果，适合做程序二次处理或继续清洗。

### `bilibili_search_result.xlsx`

表格版结果，适合直接筛选、查看、汇报。

## 运行逻辑

项目的真实工作流是：

1. 读取配置关键词
2. 根据开始和结束日期拆成每日时间段
3. 对每个关键词按每天区间执行搜索
4. 每个区间分别跑 `最新发布` 和 `最多播放`
5. 从搜索接口持续翻页直到无新数据
6. 结果按 `bvid/url/aid` 去重后写入 JSON
7. 全量导出 XLSX
8. 休眠到下一个执行周期

这套流程的优势非常直接：

- 时间边界清楚
- 结果沉淀稳定
- 长时间运行不容易乱
- 很适合做持续监控

## 为什么这个脚本不是普通搜索脚本

普通搜索脚本的问题通常很明显：

- 只抓一页
- 只抓一种排序
- 没有时间切分
- 没有定时能力
- 没有累计沉淀
- 没有去重
- 抓完一轮就结束

而这个项目做的是另一件事：

> 它在把 B站搜索结果，变成一个可以持续追踪、持续沉淀、持续更新的数据流。

这就是它真正有价值的地方。

---

## 注意事项

- B站搜索接口存在风控和结果上限，长时间高频抓取时要控制节奏
- `REQUEST_INTERVAL` 不要压得太低
- 某些关键词在未登录状态下结果可能受限
- 搜索接口返回的 `numPages` 并不总是可靠，脚本已改为按“空页 / 全重复 / 硬上限”终止翻页
- 若要长期运行，建议放在稳定网络环境中执行
